# Diagnosing Bugs

一个针对棘手 bug 的纪律。仅在有明确理由时才跳过阶段。

在探索代码库时，阅读 `CONTEXT.md`（如果存在）以获取相关 modules 的清晰心智模型，并检查你正在接触的区域的 ADR。

## Phase 1 — 构建反馈循环

**这是本 skill 的核心。** 其他一切都是机械操作。如果你有一个**紧密**的 pass/fail 信号来检测这个 bug ——一个可以_对这个_ bug 变红的信号——你将会找到原因；二分查找、假设验证和 instrumentation 都只是消费它而已。如果你没有，再多的代码审查也救不了你。

在这里投入不成比例的努力。**要激进。要有创意。拒绝放弃。**

### 构建反馈循环的方法——大致按此顺序尝试

1. **失败的测试** 在能触及 bug 的任意 seam —— unit、 integration、 e2e。
2. **Curl / HTTP 脚本** 针对正在运行的 dev server。
3. **CLI 调用** 使用 fixture 输入，将 stdout 与已知正确的 snapshot 进行比较。
4. **Headless browser 脚本**（Playwright / Puppeteer）——驱动 UI，对 DOM/console/network 进行断言。
5. **重放捕获的 trace。** 将真实的网络请求 / payload / 事件日志保存到磁盘；通过隔离的代码路径重放它。
6. **可丢弃的 harness。** 启动系统的最小子集（一个 service， mocked 依赖），通过单个 function 调用执行 bug 代码路径。
7. **Property / fuzz loop。** 如果 bug 是"有时输出错误"，运行 1000 个随机输入并寻找 failure mode。
8. **Bisection harness。** 如果 bug 出现在两个已知状态（commit、 dataset、 version）之间，自动化"在状态 X 启动，检查，重复"，以便你可以运行 `git bisect run`。
9. **差分循环。** 对旧版本 vs 新版本（或两种配置）运行相同的输入并比较输出。
10. **HITL bash 脚本。** 最后手段。如果人类必须点击，用 `scripts/hitl-loop.template.sh` _驱动他们_，使循环仍然有结构。捕获的输出反馈给你。

构建正确的反馈循环， bug 就完成了 90%。

### 收紧密闭环

将循环视为一个产品。一旦你有了_一个_循环，**收紧密闭它**：

- 我能让它更快吗？（缓存设置、跳过无关的初始化、缩小测试范围。）
- 我能让信号更清晰吗？（针对具体症状进行断言，而不是"没有崩溃"。）
- 我能让它更确定性吗？（固定时间、种子 RNG、隔离文件系统、冻结网络。）

一个 30 秒的 flaky 循环几乎不比没有循环好；一个 2 秒的确定性循环是 tight ——是调试的超能力。

### Non-deterministic bugs

目标不是干净的复现而是**更高的复现率**。循环触发 100 次、并行化、增加压力、缩小时间窗口、注入 sleeps。50% flaky 的 bug 是可调试的；1% 则不是——不断提高复现率直到它可调试。

### 当你真的无法构建循环时

停下来并明确说出来。列出你尝试过的。向用户请求：(a) 访问任何可以复现它的环境，(b) 捕获的 artifact （HAR 文件、 log dump、 core dump、带时间戳的屏幕录制），或 (c) 添加临时生产 instrumentation 的权限。**不要**在没有循环的情况下继续进行假设。

### 完成标准——一个变红的紧密循环

Phase 1 完成的标准是循环**紧密**且**能变红**：你可以命名**一个命令**——一个脚本路径、一个测试调用、一个 curl ——你已经**至少运行过一次**（粘贴调用及其输出），并且该命令：

- [ ] **能变红**——它驱动实际的 bug 代码路径并断言**用户的具体症状**，因此它可以因为这个 bug 变红，并在修复后变绿。不仅仅是"运行不报错"——它必须能够_捕获这个具体的 bug_。
- [ ] **确定性**——每次运行相同的结果（flaky bugs：固定的高复现率，见上文）。
- [ ] **快速**——秒级，而不是分钟级。
- [ ] **可由 Agent 运行**——可以在无人值守的情况下运行；只有在通过 `scripts/hitl-loop.template.sh` 时才有人类在循环中。

如果你发现自己在这个命令存在之前就开始阅读代码构建理论，**停下来——直接跳到假设正是这个 skill 要阻止的失败。** 没有能变红的命令，就没有 Phase 2。

## Phase 2 — 复现 + 最小化

运行循环。看到它变红—— bug 出现。

确认：

- [ ] 循环产生了**用户**描述的 failure mode ——而不是恰好附近的另一个不同失败。错误的 bug = 错误的修复。
- [ ] 失败在多次运行中可重现（或者，对于 non-deterministic bugs，以足够高的速率重现以进行调试）。
- [ ] 你已经捕获了具体的症状（错误消息、错误输出、慢速度），以便后续阶段可以验证修复是否真正解决它。

### 最小化

一旦它变红，将复现缩小到**仍然变红的最小场景**。**一次一个**地削减输入、调用者、配置、数据和步骤，每次削减后重新运行循环——只保留对失败起作用的内容。

为什么值得这么做：一个最小的复现缩小了 Phase 3 中的假设空间（留下更少的可怀疑部件），并成为 Phase 5 中的干净的 regression test。

完成标准：**每个剩余元素都是起作用的**——移除其中任何一个都会使循环变绿。

在复现**并且**最小化之前不要继续。

## Phase 3 — 提出假设

在测试任何假设之前，生成 **3-5 个排好序的假设**。单一假设生成会锚定在第一个看似合理的想法上。

每个假设必须是**可证伪的**：陈述它所做的预测。

> 格式："如果 <X> 是原因，那么 <改变 Y> 将使 bug 消失 / <改变 Z> 将使 bug 更严重。"

如果你无法陈述预测，这个假设就是一个 vibe ——放弃它或精炼它。

**在测试之前向用户展示排序列表。** 他们通常拥有可以立即重新排序的领域知识（"我们刚刚对 #3 部署了一个变更"），或者知道他们已经排除了哪些假设。便宜的检查点，省时的大户。如果不阻塞——如果用户不在，就按你的排序继续。

## Phase 4 — 仪表化

每个探测必须映射到来自 Phase 3 的特定预测。**一次只改变一个变量。**

工具偏好：

1. **Debugger / REPL 检查** 如果环境支持。一个断点胜过十行日志。
2. **定向日志** 在能区分假设的边界处。
3. 永远不要"记录一切然后 grep"。

**给每条调试日志打标签**，加上唯一前缀，例如 `[DEBUG-a4f2]`。最后的清理就变成了一个简单的 grep。未标记的日志保留；标记的日志删除。

**性能分支。** 对于性能回归，日志通常是错的。取而代之：建立基线测量（time harness、`performance.now()`、 profile、查询计划），然后二分查找。先测量，后修复。

## Phase 5 — 修复 + Regression test

在**修复之前**编写 regression test ——但只有在有一个**正确的 seam** 时才这样做。

正确的 seam 是指测试在调用点处实际发生的方式下**执行真正的 bug 模式**的 seam。如果唯一可用的 seam 太浅（当 bug 需要多个调用者时是单调用者测试，无法复制触发 bug 的链的 unit test），在那里的 regression test 会给出错误的信心。

**如果没有正确的 seam 存在，这本身就是一个发现。** 记录下来。代码库架构阻止了 bug 被锁定。将此标记给下一阶段。

如果正确的 seam 存在：

1. 将最小化的复现转换为该 seam 上的失败测试。
2. 看到它失败。
3. 应用修复。
4. 看到它通过。
5. 对原始（未最小化的）场景重新运行 Phase 1 反馈循环。

## Phase 6 — 清理 + 事后分析

在宣布完成之前必须完成：

- [ ] 原始复现不再复现（重新运行 Phase 1 循环）
- [ ] Regression test 通过（或没有 seam 的情况已记录）
- [ ] 所有 `[DEBUG-...]` instrumentation 已移除（用 grep 匹配前缀）
- [ ] 可丢弃的 prototypes 已删除（或移动到明确标记的调试位置）
- [ ] 最终正确的假设已写在 commit / PR 消息中——这样下一个调试者就能学到

**然后问：什么本来可以防止这个 bug？** 如果答案涉及架构变更（没有好的测试 seam、调用者纠缠不清、隐藏的耦合），将具体情况传递给 `/improve-codebase-architecture` skill。在**修复之后**才提出建议，而不是之前——你现在拥有的信息比开始时更多。
