# UI Prototype

在单个路由上生成**几种截然不同的 UI 变体**，从浮动的底部栏切换。用户在浏览器中切换变体，选择一个（或从每个中偷取一部分），然后丢弃其余部分。

如果问题是关于逻辑/状态而不是某事物看起来什么样——错误的分支。使用 [LOGIC-zh.md](LOGIC-zh.md)。

## 何时这是正确的形态

- "这个页面应该看起来什么样？"
- "我想在承诺之前看到这个 dashboard 的几个选项。"
- "尝试设置屏幕的另一种布局。"
- 任何时候用户本来会花一天时间在他们脑海中的三个模糊的 mockups 之间选择。

## 两种子形态——强烈偏好子形态 A

当一个 UI prototype **与应用的其他部分对接**时——真实的 header、真实的 sidebar、真实的数据、真实的密度——它更容易评判。一个孤立的可丢弃路由是一个真空：每个变体在隔离中看起来都不错。每当有一个合理的现有页面来托管变体时，默认使用子形态 A。只有当 prototype 真的没有附近的家时，才使用子形态 B。

### 子形态 A ——对现有页面的调整（首选）

路由已经存在。变体在**同一路由**上渲染，由 `?variant=` URL search param 控制。现有的数据获取、 params 和 auth 都保留——只有渲染切换。这是默认选项；除非有特定的理由不这样做。

如果 prototype 是为了还没有页面但*自然会存在于某个页面内*的东西（dashboard 的新 section、设置屏幕上的新 card、现有 flow 中的新步骤）——那仍然是子形态 A。将变体挂载在宿主页面内。

### 子形态 B ——新页面（最后手段）

只有当正在设计的对象确实没有现有的页面可以放置在内时才使用此选项——例如，一个全新的顶层表面，或一个无法嵌入到任何合理位置的 flow。

创建一个**可丢弃的路由**，遵循项目已使用的路由约定——不要发明新的顶层结构。命名时使其明显是一个 prototype （例如在路径或文件名中包含 `prototype` 这个词）。相同的 `?variant=` 模式。

在承诺使用子形态 B 之前，理智检查：真的没有现有的页面可以嵌入吗？空白的路由隐藏了有内容的路由会暴露的设计问题。

在两种子形态中，浮动的底部栏是相同的。

## 流程

### 1. 陈述问题并选择 N

默认 **3 个变体**。超过 5 个就不再是截然不同而是开始成为噪音——在那里设置上限。

用一句话写下计划，放在 prototype 的位置或文件顶部的注释中：

> "设置页面的三个变体，通过 `?variant=` 切换，在现有的 `/settings` 路由上。"

无论用户是否在场可以提出异议，这都适用。

### 2. 生成截然不同的变体

起草每个变体。让每个变体坚持：

- 页面的目的和它有权访问的数据。
- 项目的 component library / 样式系统（TailwindCSS、 shadcn、 MUI、 plain CSS，随便什么）。
- 一个清晰的导出 component 名称，例如 `VariantA`、`VariantB`、`VariantC`。

变体必须在**结构上不同**——不同的布局、不同的信息层次、不同的主要操作，而不仅仅是不同的颜色。三个略微调整的卡片网格不是 UI prototype，而是 wallpaper。如果两个草稿过于相似，用明确的"不要使用卡片网格"指导重新做一个。

### 3. 将它们连接在一起

在路由上创建一个单一的 switcher component：

```tsx
// 伪代码——适应项目的 framework
const variant = searchParams.get('variant') ?? 'A';
return (
 <>
 {variant === 'A' && <VariantA {...data} />}
 {variant === 'B' && <VariantB {...data} />}
 {variant === 'C' && <VariantC {...data} />}
 <PrototypeSwitcher variants={['A','B','C']} current={variant} />
 </>
);
```

对于子形态 A （现有页面）：保持 switcher 之上的所有现有数据获取；只有每个变体的渲染子树发生变化。

对于子形态 B （新页面）：`/prototype/<name>` 下的可丢弃路由挂载相同的 switcher。

### 4. 构建浮动切换器

屏幕底部中央的一个小型固定位置栏，包含三个部分：

- **左箭头**——循环到前一个变体（环绕）。
- **变体标签**——显示当前变体键，如果变体导出一个名称，也显示该名称。例如 `B — Sidebar layout`。
- **右箭头**——向前循环（环绕）。

行为：

- 点击箭头更新 URL search param （使用 framework 的路由器—— Next 上的 `router.replace`， React Router 上的 `navigate` 等），以便变体可分享且在重载后稳定。
- 键盘：`←` 和 `→` 箭头键也循环。当 `<input>`、`<textarea>` 或 `[contenteditable]` 聚焦时，不拦截箭头键。
- 在视觉上与页面区分开来（例如高对比度药丸、微妙的阴影），以便明显不是正在评估的设计的一部分。
- 在生产构建中隐藏——通过 `process.env.NODE_ENV !== 'production'` 或等效检查进行门控，以便意外的 prototype 合并不会将栏发送给用户。

将切换器放在一个单一的共享 component 中，以便两种子形态都可以复用。将其放在项目中共享 UI 所在的位置。

### 5. 交付

暴露 URL （以及 `?variant=` 键）。用户随时会切换查看。有趣的反馈通常是**"我想要 B 的 header 和 C 的 sidebar"**——这才是他们真正想要的设计。

### 6. 捕获答案并清理

一旦某个变体胜出，写下是哪个以及为什么（commit message、 ADR、 issue，或如果正在 AFK 运行且用户尚未响应，则放在 prototype 旁边的 `NOTES.md` 中）。然后：

- **子形态 A**——删除失败的变体和切换器；将胜出的变体融入现有页面。
- **子形态 B**——将胜出的变体提升为真实路由，删除可丢弃路由和切换器。

不要将变体 components 或切换器遗留。它们迅速腐烂并混淆下一个读者。

## 反模式

- **仅在颜色或文案上不同的变体。** 那是调整，不是 prototype。真正的变体在结构上有分歧。
- **在变体之间共享太多代码。** 共享 `<Header>` 是可以的；共享 `<Layout>` 则破坏了目的。每个变体应该能够自由地抛弃布局。
- **将变体连接到真实的 mutations。** 只读 prototypes 是可以的。如果一个变体需要进行 mutation，将其指向一个 stub ——问题是"这应该看起来什么样"，而不是"backend 是否工作"。
- **将 prototype 直接提升到生产环境。** 变体代码是在 prototype 约束下编写的（没有测试、最小的错误处理）。在将其融入时正确地重写它。
