# Triage

通过 triage roles 的小型 state machine 推动项目 issue tracker 上的 issues。

如果此 repo 将外部 pull requests 视为请求面（参见 issue-tracker config）， triage 也涵盖它们：**PR 是带有代码的 issue**——相同的 roles、相同的 states、相同的 machine，下面注明了一些"for a PR"的 delta。根据 tracker config 将裸 `#42` 解析为 issue 或 PR。

Triage 期间发布到 issue tracker 的每条 comment 或 issue **必须**以此免责声明开头：

```
> *This was generated by AI during triage.*
```

## Roles

两个**类别** roles：

- `bug` ——有东西坏了
- `enhancement` ——新功能或改进

五个**状态** roles：

- `needs-triage` ——维护者需要评估
- `needs-info` ——等待报告者提供更多信息
- `ready-for-agent` ——完全指定， AFK agent 可拾取
- `ready-for-human` ——需要人类实现
- `wontfix` ——不会被执行

对于 PR，相同 states 针对附带的代码解读：`ready-for-agent` 表示已附上 brief， agent 应对 diff 进行下一步操作；`ready-for-human` 表示准备就绪，可由人类合并。

每个经 triage 处理的 issue 应携带恰好一个类别 role 和一个状态 role。如果状态 roles 冲突，标记它并在做其他任何事情之前询问维护者。

这些是规范角色名称—— issue tracker 中使用的实际 label 字符串可能不同。映射应该已提供给你——如果没有，运行 `/setup-matt-pocock-skills`。

状态转换：一个未标记的 issue 通常先进入 `needs-triage`；从那里它移动到 `needs-info`、`ready-for-agent`、`ready-for-human` 或 `wontfix`。`needs-info` 在报告者回复后返回到 `needs-triage`。维护者可以随时覆盖——标记看起来不寻常的转换并在继续之前询问。

## 调用

维护者调用 `/triage` 并用自然语言描述他们想要的内容。解释请求并执行。示例：

- "Show me anything that needs my attention"
- "Let's look at #42"（issue 或 PR）
- "Move #42 to ready-for-agent"
- "What's ready for agents to pick up?"

## 显示需要关注的内容

查询 issue tracker 并呈现三个桶，最早的最先：

1. **未标记** ——从未被 triage。
2. **`needs-triage`** ——评估进行中。
3. **`needs-info` 且报告者自上次 triage 笔记后有活动** ——需要重新评估。

当 PRs 在范围内时，将这些桶中的外部 PRs 纳入，并标记每行 `[PR]` 或 `[issue]`。 Discovery 仅暴露*外部* PRs （tracker config 定义谁算作外部）——协作者进行中的 PR 不是 triage 工作。此过滤器仅用于 discovery；明确命名的 PR 无论作者如何总是被 triage。

显示计数和每项的一行摘要。让维护者选择。

## Triage 特定的 issue 或 PR

1. **收集 context。** 阅读完整的 issue 或 PR （body、 comments、 labels、作者、日期；对于 PR，还有 diff）。解析任何先前的 triage 笔记，这样你就不会重新询问已解决的问题。使用项目的 domain glossary 探索代码库，尊重该区域的 ADR。针对代码库运行两个检查：(a) **冗余** ——按领域概念（不仅仅是请求的措辞）搜索请求行为的现有实现，并报告你查看了哪里。如果找到，就是已经实现的 `wontfix`（步骤 5）。(b) **先前的拒绝** ——读取 `.out-of-scope/*.md` 并找出任何与此请求相似的内容。

2. **推荐。** 告诉维护者你的类别和状态推荐及其理由，加上与请求相关的简短代码库摘要——包括是否已实现。等待指示。

3. **验证声明。** 在任何 grilling 之前，检查声明是否成立。对于 bug，根据报告者的步骤复现它。对于 PR，确认 diff 做了它声称的事情—— checkout 它，运行相关的 tests 或 commands。报告发生了什么：已确认（带有代码路径）、失败或不充分的细节（一个强烈的 `needs-info` 信号）。一个已确认的验证会产生更强大的 agent brief。

4. **Grill （如果需要）。** 如果请求需要充实，一起运行 `/grilling` 和 `/domain-modeling` skills ——一次一个问题地将其 grilling 成形状，当决策落地时内联精炼领域术语并更新 `CONTEXT.md`/ADR。

5. **应用结果：**
 - `ready-for-agent` ——发布 agent brief comment （[AGENT-BRIEF-zh.md](AGENT-BRIEF-zh.md)）。
 - `ready-for-human` ——与 agent brief 相同的结构，但说明为什么不能委托（判断性决策、外部访问、设计决策、手动测试）。
 - `needs-info` ——发布 triage 笔记（下面是模板）。
 - `wontfix` ——关闭， comment 取决于*为什么*：
 - **Already implemented** ——变更已存在于代码库。指向它所在的位置；**不要**写入 `.out-of-scope/`（该 KB 用于*被拒绝的*请求，而不是已构建的）。
 - **Rejected (bug)** ——礼貌的解释，然后关闭。
 - **Rejected (enhancement)** ——写入 `.out-of-scope/`，从 comment 链接到它，然后关闭（[OUT-OF-SCOPE-zh.md](OUT-OF-SCOPE-zh.md)）。
 - `needs-triage` ——应用 role。如果有部分进展，可选 comment。

## 快速状态覆盖

如果维护者说"move #42 to ready-for-agent"，信任他们并直接应用 role。确认你将要做的事情（role 变更、 comment、关闭），然后行动。跳过 grilling。如果在没有 grilling session 的情况下移动到 `ready-for-agent`，询问他们是否想编写一个 agent brief。

## Needs-info 模板

```markdown
## Triage Notes

**我们已经确定的内容：**

- 要点 1
- 要点 2

**我们仍然需要你提供的内容（@reporter）：**

- 问题 1
- 问题 2
```

在"已确定"下捕获 grilling 期间解决的所有内容，以便工作不会丢失。问题必须具体且可操作，而不是"请提供更多信息"。

## 恢复先前的 session

如果 issue 或 PR 上存在先前的 triage 笔记，读取它们，检查报告者是否已回答任何未解决的问题，并在继续之前呈现更新的画面。不要重新询问已解决的问题。
