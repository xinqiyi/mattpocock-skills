# ADR 格式

ADR 位于 `docs/adr/` 目录下，使用顺序编号：`0001-slug.md`、`0002-slug.md` 等。

懒加载创建 `docs/adr/` 目录——只有在第一个 ADR 需要时才创建。

## 模板

```md
# {决策的简短标题}

{1-3 句话：上下文是什么，我们决定了什么，以及为什么。}
```

就是这样。一个 ADR 可以只是一个段落。价值在于记录*做了*决定以及*为什么*——而不是填写各个 section。

## 可选 Sections

仅当这些能带来真正的价值时才包含它们。大多数 ADR 不需要它们。

- **Status** frontmatter （`proposed | accepted | deprecated | superseded by ADR-NNNN`）——当决策被重新审视时有用
- **Considered Options**——仅当被拒绝的替代方案值得记住时
- **Consequences**——仅当非显而易见的后续影响需要被指出时

## 编号

扫描 `docs/adr/` 获取最高的现有编号并加一。

## 何时提供 ADR

以下三者必须都为真：

1. **难以逆转**——事后改变想法的成本是显著的
2. **没有 context 会令人惊讶**——未来的读者会看到代码并想知道"他们到底为什么这样做？"
3. **是真正权衡的结果**——有真正的替代方案，你出于特定原因选择了其中一个

如果一个决定很容易逆转，跳过它——你只需要逆转它。如果它不令人惊讶，没有人会想知道为什么。如果没有真正的替代方案，除了"我们做了显而易见的事情"之外没什么可记录的。

### 什么符合条件

- **架构形态。** "我们使用 monorepo。""Write model 是 event-sourced 的， read model 投影到 Postgres。"
- **Context 之间的集成模式。** "Ordering 和 Billing 通过 domain events 通信，而不是同步 HTTP。"
- **带来锁定效应的技术选择。** 数据库、消息总线、 auth provider、部署目标。不是每个 library ——只是那些更换需要四分之一个季度的。
- **边界和范围决策。** "Customer 数据由 Customer context 拥有；其他 contexts 仅通过 ID 引用它。"明确的否定与肯定同样有价值。
- **故意偏离显而易见的路径。** "我们因为 X 原因使用手动 SQL 而不是 ORM。"任何合理读者会认为相反的情况。这些阻止下一位 engineer "修复"某些本是故意的设计。
- **代码中不可见的约束。** "由于合规要求，我们不能使用 AWS。""由于 partner API 合同，响应时间必须在 200ms 以下。"
- **被拒绝的替代方案，如果拒绝理由不显而易见。** 如果你考虑了 GraphQL 并因为微妙的原因选择了 REST，记录下来——否则六个月内会有人再次建议 GraphQL。
