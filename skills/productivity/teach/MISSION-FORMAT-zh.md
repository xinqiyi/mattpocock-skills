# MISSION.md 格式

`MISSION.md` 位于 workspace 根目录。它捕获用户学习该主题的_原因_。每个教学决策——下一步教什么、呈现哪些资源、设计哪些练习——都应追溯到本文档。

## 模板

```md
# Mission: {Topic}

## Why
{1-3 句话。用户追求的具体现实世界目标。当他们拥有此技能时，他们的生活或工作会发生什么变化？避免抽象表述如"为了理解 X"——推动底层结果。}

## Success looks like
- {用户将能做的具体、可观察的事情}
- {另一个具体的事情}
- {…}

## Constraints
- {时间、预算、先前承诺、学习偏好、任何限制方法的东西}

## Out of scope
- {用户明确不想现在追求的相邻主题——保护最近发展区}
```

## 规则

- **每个 workspace 一个 mission。** 如果用户想学两个不相关的东西，那是两个 workspaces。
- **具体胜过抽象。** "十月前跑一个半程马拉松"胜过"变得更健康。""向我的团队交付一个 Rust CLI"胜过"学习 Rust。"
- **对模糊说"不"。** 如果用户无法表达为什么，在写任何东西之前先访谈他们。坏的 mission 比没有 mission 更糟。
- **当现实变化时修订。** Missions 会变化。当用户的目标移动时，更新此文件——不要留下过时的 mission 指导未来的 sessions。
- **保持简短。** 如果 `MISSION.md` 超过一个屏幕，它就不再是指南针而变成了计划。
